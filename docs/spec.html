Introduction

This document, the Enterprise Ethereum Alliance Enhanced BFT Specification, defines enhancements to the BFT algorithm including liveness and security.

Protocol Overview

The actors in the system are: - Validator - multiple nodes that participate in the consensus protocol to finalise blocks. - Proposer - The proposer is a Validator that proposes a block. Only one node acts in this role for any given round. - Standard node - a node which does not propose or validate blocks. The finalization protocol described below is executed for each block height with the objective of creating a block at that block height. The finalization protocol is organised into rounds starting from round zero. The operation of the finalization protocol is described as follows. Proposer The Proposer generates the block in ethereum format and sends a Proposal message as a mutlicast to all other Validators. When using Gossip, the multicast message is sent to all nodes. It then behaves as a validator. Validator A Validator awaits a Proposal message from the Proposer for the round. When a Proposal message is received, the Validator makes a number checks to determine if the Proposal message is valid. If not valid, it ignores the Proposal message and awaits another Proposal message. If a Validator receives a valid Proposal message containing a valid block, it generates a Prepare message for the same round and height containing the hash of the block included in the Proposal message and sends this via multicast to all other Validators. The Validator then waits for Prepare messages from the other Validators. When Prepare messages matching the Proposal message have been received from at least two thirds minus one of the Validators, the Validator generates a Commit message for the same round and height containing the hash of the block included in the Proposal message and the Commit seal and sends this via multicast to all other Validators. The Validator then waits for Commit messages from the other Validators. When Commit messages matching the Proposal message have been received from at least two thirds of the Validators, the Validator goes ahead and imports the block to the chain. The Validator also adds the received Commit seals to the block to allow any other node to verify the block was produced correctly according to the protocol. Following successful addition of the block to the chain, a new Proposer is determined. The new Proposer then repeats the above process for the next block height. If the Validator detected a timeout before two thirds of the Commit messages were received, the Validator generates a Round Change message with sufficient information to safely move to the next round thus preventing a fork from being possible. The protocol then repeats as described above in this section.

Message Structures

This section defines the structures of the "BFT" sub-protocol messages communicated between ethereum nodes. All messages MUST be encoded using RLP encoding serialisation as specified in [https://github.com/ethereum/wiki/wiki/RLP]. For any type of message, the signature corresponds to the hash over the message using the validator private key: hashForMessageSignature(message) = KEC(concatenate(message.ID, RLP(message.payload)))

Proposal

The block proposer must send the Proposal message to all other validators at the beginning of any round. **Message content** The Proposal message contains the following fields: - `0x00` DATA, x? bytes – The message id for the Proposal message type. - `payload` – A list containing each of the following: - `height` QUANTITY, 8 bytes – The height of the block being proposed, as an integer. - `round` QUANTITY, 4 bytes – The round number, as an integer. - `digest` DATA, 32 bytes – The Keccak hash of the proposed block. - `signature` DATA, 65 bytes – The signature of the payload. - `proposedBlock` – The content of the proposed block in ethereum block format.

Prepare

The Prepare message MUST be sent when a validator that is not the designated block proposer for a round receives a Proposal message. **Message content** The Prepare message contains the following fields: - `0x01` DATA, x? bytes – The message id for the Prepare message type. - `payload` – A list containing each of the following: - `height` QUANTITY, 8 bytes – The height of the block being prepared, as an integer. - `round` QUANTITY, 4 bytes – The round number, as an integer. - `digest` DATA, 32 bytes – The digest of the prepared block? - `signature` DATA, 65 bytes – The signature of the payload.

Commit

The Commit message MUST be sent by a validator when "enough" Prepare messages matching the Proposal sent by the designated proposer mof the round are received. **Message content** The Commit message contains the following fields: - `0x02` DATA, x? bytes – The message id for the Commit message type. - `payload` – An array of one or more of the following: - `height` QUANTITY, 8 bytes – The height of the block being prepared, as an integer. - `round` QUANTITY, 4 bytes – The round number, as an integer. - `digest` DATA, 32 bytes – The digest of the prepared block? - `commitSeal` DATA, 32 bytes – The signature of the validator over the proposed block included in the Proposal message received by the designated proposer of the round. - `signature` DATA, 65 bytes – The signature of the payload.

Round Change

A Round Change message MUST be sent by a validator when they suspect that no agreement on the block to be added to the blockchain with the given height will be reached in this round. **Message content** The Round Change message contains the following fields: - `0x03` DATA, x? bytes – The message id for the Round Change message type. - `payload` – An array of one or more of the following: - `height` QUANTITY, 8 bytes – The height of the block for which the validator is trying to reach agreement, as an integer. - `round` QUANTITY, 4 bytes – The round number to which the validator intends to move to, as an integer. - `preparedCertificate` DATA – See below. - `signature` DATA, 65 bytes – The signature of the concatenation of the message id and the payload. - `latestProposedBlock` DATA - `nil` or the content of the latest prepared block in ethereum block format. Should be `nil` if preparedCertificate is nil. The `preparedCertificate` may be `nil` or a list of: - `signedPartOfProposalMessage` – The payload and signature copied from the latest prepared Proposal message. - `signedPartOfPrepareMessages` – The list of payloads and signatures copied from the latest prepared Prepare messages. If `latestProposedBlock` is not nil, then the digest of `signedPartOfProposalMessage` and all messages included in `signedPartOfPrepareMessages` should match the Keccack hash of `proposedBlock`.

Protocol Description

The Enhanced BFT block finalization protocol consists of a series of messages exchanged by the validators using the gossip protocol to reach consensus on block finalization. All of the Enhanced BFT messages described below are always signed by the sender. Initially, the selected block proposer sends a Proposal message (`PROPOSAL_MSG_CODE, r, s, PB`) to all other validators, where: - `r` is the current round (initially 0) of the Enhanced BFT consensus protocol - `s` is the height (called the sequence in Enhance BFT terminology) of the proposed block - `PB` is the proposed block with the `IstanbulData.Seal` (proposer seal) calculated as described in Proposer seal calculation. When a validator receives a Preposer message for round 0 and sequence `s`, it broadcasts a Prepare message (`PREPARE_MSG_CODE, r = 0, s, Hash(PB)`) to all other validators. A validator is said to be prepared on round `r`, sequence `s`, and proposed block `PB` if it had received (`QUORUM(NV)-1`) Prepare messages with the propose-block-hash matching the hash of the proposed block `PB` included in the Proposal message received previously. When a validator `v` is prepared, it broadcasts a Commit message (`COMMIT_MSG_CODE, r, s, Hash(PB), Commit-Seal`) to all other validators where `Commit-Seal` is calculated as described in Committed seal calculation. When a validator receives `QUORUM(NV)` commit messages for the same `r`, `s`, and with the proposed-block-hash matching the hash of the proposed block `PB` received initially as part of the Proposal message, it adds the `QUORUM(NV)` received `Commit-Seal` values to the `IstanbulExtra.CommittedSeals` field and sends the block to the other validators, including itself. However, a Byzantine block proposer could either never propose a valid block or propose different blocks to different validators impeding the Enhanced BFT finalization protocol from reaching consensus on the next block to add to the blockchain. To solve this liveness issue, every time a validator receives a new valid block, it starts a new round timer. When the round timer for a validator expires, the validator broadcasts a Round-Change message (`ROUND_CHANGE_MSG_CODE, r+1, s, Proposal/Prepare-Certificate`) to all validators. The content of the Proposal/Prepare-Certificate is empty if the validator is not prepared on any proposed block for sequence `s`, otherwise the Proposal/Prepare-Certificate contains the set of Proposal and Prepare messages for the highest round at which validator `v` is prepared for sequence `s` on a valid proposed block `PB`. For example, if validator `v` prepared on proposed block `PB1` at round 0, it did not prepare on any block at round 2, it prepared on proposed block `PB2` at round 3, and it did not prepare on any block at round 4, then the Proposal/Prepare-Certificate included in the next Round-Change message includes the Proposal and Prepare messages that caused validator `v` to be prepared at round 3 on proposed block `PB2`. When the proposer for round `r`+1 receives `QUORUM(NV)` Round-Change messages for the same `r`+1 round and sequence `s`, it broadcasts a New-Round message (`NEW_ROUND_MSG_CODE, r+1, h, Round-Change-Certificate, Proposal-message`) to all validators. The Round-Change certificate contains the `QUORUM(NV)` Round-Change messages received and the `Proposal-message` field contains the Proposal message for round `r`+1. The Proposal message is determined in the following way. If the proposer for round `r`+1 has received at least 1 Round-Change message with a valid and non-empty Preposal/Prepare-Certificate, then the Proposal message contains the Proposal message included in Proposal/Prepare-Certificate with the highest round number. Otherwise, the Proposal message is a Proposal message signed by the validator for any valid block. When a validator receives a New-Round message, it verifies that the `Round-Change-Certificate` field contains `QUORUM(NV)` valid Round-Change messages and then verifies the Proposal message included in the New-Round message by performing a computation similar to the one used by the sender of the New-Round message. If no Round-Change message included in the Round-Change-Certificate includes a Proposal/Prepare certificate, then any Proposal message for a valid block included in the New-Round message is accepted. If validation of the New-Round message is successful, the validator will act as if it has received the Proposal message from the proposer and hence will broadcast a Prepare message as described above.
